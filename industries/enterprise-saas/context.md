# Industry Context: Enterprise SaaS

## Domain Characteristics
- Two distinct users: the buyer (who decides) and the user (who does the work)
- Onboarding must deliver value before the trial ends
- Power users and casual users coexist — both must be served
- IT/Security/Compliance are invisible stakeholders who block adoption

## Key User Mental Models
- "Will this fit into how we already work?" (integration anxiety)
- Power users: "Is there a faster way?" — always hunting for shortcuts
- Occasional users: "Where was that again?" — navigation memory is weak
- Admins: "Who has access to what?" — control and auditability
- "My old tool did it this way" — migration pain is real

## Common UX Anti-Patterns (avoid)
- Hiding advanced features behind discovery by accident (not intentional progressive disclosure)
- No empty states with onboarding prompts
- Permission errors with no explanation of who to contact
- Too many notification types with no control
- Exporting data requires IT help
- No keyboard shortcuts for power users
- Inconsistent terminology across the product

## High-Value UX Patterns
- Role-based onboarding (admin vs. end user vs. viewer)
- In-app tooltips and contextual help (not just a help doc link)
- Bulk actions for productivity at scale
- Customizable dashboards / saved views
- Audit trail and activity history
- Keyboard shortcut layer (power user retention)
- Inline editing (reduces context switching)
- Smart defaults that work for most teams out of the box

## Key Metrics for Enterprise SaaS UX
- Time to first value (TTFV) in trial
- Feature adoption rate by persona
- Daily/Weekly active users (DAU/WAU)
- Support ticket volume by feature area
- NPS by role (admin vs. end user)
- Seats activated vs. seats paid (license utilization)

## Competitive Reference Points
- Notion: Flexible structure, high learning curve trade-off
- Linear: Speed-first issue tracking UX
- Figma: Real-time collaboration patterns
- Salesforce: Dense enterprise UI, role customization
- Slack: Notification control patterns, workspace model
- Jira: Power vs. complexity management
